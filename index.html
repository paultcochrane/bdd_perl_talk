What is Behaviour Driven Development?

  • Comes from the Agile concept of User Stories
  • A form of requirements specification
  • Allows business stakeholders to define features in the language of their domain
  • Automated acceptance testing
  • Outside-in development
  • Complements and uses Test Driven Development (TDD)
  • Not a replacement for TDD!

Cucumber

  • Stakeholders/Domain experts write requirements in business-readable domain-specific language.
  • This can be parsed and the underlying (matching) tests get executed.
  • Serves as documentation for the desired features.
  • Domain experts don't have to write code!
  • Domain experts and developers understand what is to be built.
  • Parseable natural language: Gherkin 

            Feature User login
                As a user of the system
                In order to see my account information
                I want to be able to log in

                Scenario Successful login
                    Given I have an account on the system
                    When I enter my username and password
                    Then I should be presented with my account information

Gherkin the language


Feature name of current feature
    # description of feature; purely for documentation
    In order to [business value]
    As a [role]
    I want to [some action]

    # possible background information, relevant to all scenarios
    Background
        Given [context/state]

    # scenario(s) describing details of feature
    Scenario name of current scenario
        Given [context/state]
        When [action]
        Then [outcome]

    # loop over a list of scenario examples
    ScenarioOutline name of scenario outline
        Given, When, Then, etc.

        Examples
            | item header | value header |
            | item        | value        |

BDD development cycle

  • Start a feature
  • Write a scenario describing an aspect of that feature
  • Write the steps associated with that scenario
    • Each Given, When, Then is a step
  • Run pherkin
  • Steps are undefined
  • Write automation code in unit-testing framework, e.g. Test::More
  • Steps fail
  • Red -> Green -> Refactor until step passes
  • Move on to next step
  • Lather, rinse, repeat

Required directory structure for Cucumber

project/
    features/
        my_cool_feature.feature
        step_definitions/
            cool_steps.pl
            other_steps.pl
  • .feature file is written in Gherkin
  • step definition files (*\_steps.pl) are written in Perl
  • step definition files use your favourite testing framework (e.g. Test::More)
  • Recommended minimum Perl code in a step definition file: 

use strict;
use warnings;

use Test::More;
use Test::BDD::Cucumber::StepFile;
use Method::Signatures;
